title: Concept First, Code Later
category: DSA
---
I am a complete beginner at data structures and algorithms. I want to understand [data structure or algorithm — e.g. stack / queue / linked list / binary search] before touching any code.

Do NOT show me code yet.

First help me understand:
1. What problem did humans have that made them invent this?
2. What is it — explained like a real-world analogy (not a textbook)
3. How does it behave? What can you do with it and what can't you do?
4. Where does it show up in real life or in software I already use?
5. Why would I choose this over something simpler?

After I feel like I understand it:
- Ask me to explain it back in my own words
- Correct any misunderstanding gently
- Then ask if I'm ready to see how it looks in code

Keep the tone curious and encouraging. I'm learning this for the first time.

---
title: Visualize It — See the Structure
category: DSA
---
Help me visualize how [data structure or algorithm — e.g. stack / queue / binary tree / bubble sort] works.

Do NOT just describe it in words. Show me:
1. A simple ASCII diagram or step-by-step visual of the structure
2. What happens when I add something to it — show each step
3. What happens when I remove something — show each step
4. How it looks when it's empty vs full vs in the middle of an operation

Then walk me through a tiny example with real values — like numbers or names — so I can see it move.

After the visual:
- Ask me to describe what I see in my own words
- Correct any misunderstanding
- Ask if I want to see a more complex example

---
title: Explain This Algorithm Simply
category: DSA
---
Explain the [algorithm name — e.g. bubble sort / binary search / merge sort / BFS / DFS] algorithm to me like I've never heard of it.

Start with:
1. A real-world analogy — something from daily life that works the same way
2. The problem it solves — why do we need this?
3. A simple step-by-step walkthrough using a small example (use numbers or words, not code)
4. How fast or slow it is — explained simply, no math formulas yet
5. When you would use this vs when you wouldn't

Only after I understand the concept:
- Show me the simplest possible code version with clear comments
- Explain each line like I'm reading it for the first time

Keep it fun and simple. I want to feel "oh that makes sense!" not overwhelmed.

title: Solve With Me — Don't Solve For Me
category: DSA
---
I want to practice solving a DSA problem, but I need guidance — not the answer handed to me.

**Option A — I have a problem:**
[paste your problem here]

**Option B — You pick one for me:**
My current level: beginner
Topic I want to practice: [e.g. arrays / strings / loops / stacks — or just say "surprise me"]

---

However we get the problem, follow these rules:

1. Make sure I understand the problem first — ask me to explain it back in my own words
2. Ask me: "What is your first instinct? How would you approach this?"
3. If I'm stuck: give me a hint, not the answer — one small nudge at a time
4. If my approach is wrong: ask questions to help me discover why, don't just correct me
5. If my approach is right: encourage me to write it out step by step before coding
6. Only show code after I've worked through the logic myself

After we solve it:
- Explain the time and space complexity in simple terms
- Ask me if I can think of a better way
- Tell me what pattern this problem belongs to (e.g. sliding window, two pointers)

---
title: Daily DSA Warm-Up
category: DSA
---
Give me one small, beginner-friendly DSA problem to practice today.

My level: complete beginner
Language: [JavaScript / Python — pick one]

Rules:
- The problem should be solvable in 10–15 minutes
- No tricks, no advanced concepts — just clean logic practice
- Give me the problem only first — wait for me to attempt it

After I submit my answer:
- Tell me if it works and why
- Show me a clean solution with simple comments
- Explain what concept or pattern this problem practices
- Rate my solution honestly: what was good, what could be better

Keep it encouraging. The goal is to build a daily habit, not to make me feel stupid.

---
title: Pattern Recognition Trainer
category: DSA
---
I want to learn how to recognize common DSA problem patterns — so I stop feeling lost every time I see a new problem.

Today's pattern: [e.g. sliding window / two pointers / frequency counter / recursion / binary search pattern — or say "teach me the most important one for beginners"]

For this pattern:
1. Explain what kind of problem it solves — in simple words
2. Give me 2–3 example problems that use this pattern (don't solve them yet)
3. Teach me the "signal words" — what phrases in a problem hint that this pattern applies
4. Show me one simple example solved step by step using this pattern
5. Then give me a new problem and ask me: "Which pattern does this use and why?"

Wait for my answer before explaining. Help me build the instinct, not just memorize the pattern.

---
title: Rubber Duck Debug — DSA Edition
category: DSA
---
I want to explain my thinking out loud and find where my logic breaks.

Here is the problem I'm working on:
[paste the problem]

Here is my current thinking or approach (even if it's messy or incomplete):
[explain your thinking in plain English — no need for perfect code]

Your job:
- Listen to my explanation carefully
- Do NOT immediately fix it or give the answer
- Ask me questions that help me find the hole in my own logic
- If I say something unclear, ask me to explain it differently
- Only give a hint if I've been stuck after genuinely trying

After we find the issue together:
- Explain why that part of the logic was breaking
- Help me fix just that part — not rewrite everything
- Tell me what debugging habit this exercise is building

The goal is to help me think like a programmer, not to solve the problem for me.
